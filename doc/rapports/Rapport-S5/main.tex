\chapter{Introduction}

Dans le cadre de notre unité d’enseignement d’Ingénierie Logicielle, nous avons développé \textbf{WithYou}, une plateforme web collaborative permettant à plusieurs utilisateurs de regarder des vidéos YouTube ensemble de manière synchronisée et en temps réel.

Ce projet s’inspire de services existants tels que \textit{Watch2Gether}, tout en y ajoutant des fonctionnalités interactives favorisant l’engagement et la prise de décision collective, notamment un chat instantané, un système de sondage permettant de choisir la prochaine vidéo à visionner, ainsi qu’un mécanisme de notation visant à évaluer la qualité des salons et de l’expérience proposée. La plateforme propose également des salons publics et privés, accessibles via un code d’invitation unique.

L’objectif principal de \textbf{WithYou} est d’offrir une expérience de visionnage partagé fluide, conviviale et participative, dans laquelle les utilisateurs peuvent créer des salons, inviter leurs amis, échanger en temps réel et gérer collectivement une playlist de vidéos.

Ce projet nous a permis d’explorer l’ensemble du cycle de développement logiciel, depuis l’analyse des besoins jusqu’à la mise en production, en passant par la conception de l’architecture, l’implémentation de fonctionnalités temps réel via les WebSockets, ainsi que la gestion collaborative du code source à l’aide de Git et GitHub.

Nous avons adopté une méthodologie de type cascade, adaptée au cadre académique, en suivant des phases structurées (analyse, conception, développement, tests), tout en nous appuyant sur des outils modernes de gestion de projet tels que \textit{Jira} et \textit{Notion} afin d’assurer une organisation rigoureuse et une progression régulière du travail.

\chapter{Description technique}

\section{Stack technologique}

Le projet \textbf{WithYou} a été développé en s’appuyant sur une stack technologique moderne, choisie afin de garantir la performance, la fiabilité et la maintenabilité de l’application.

\subsection{Front-end : React.js et Tailwind CSS}
L’interface utilisateur est développée avec \textit{React.js}, une bibliothèque JavaScript permettant la création de composants réutilisables, dynamiques et modulaires. React facilite la gestion de l’état de l’application et la mise à jour efficace de l’interface en fonction des interactions des utilisateurs.  
Le design et la mise en forme sont assurés par \textit{Tailwind CSS}, un framework utilitaire léger favorisant un rendu visuel cohérent, propre et responsive.

\subsection{Back-end : PHP et Laravel}
La partie serveur repose sur \textit{PHP} avec le framework \textit{Laravel}, qui permet de structurer une API RESTful sécurisée. Laravel fournit nativement des outils pour la gestion des routes, la validation des requêtes, l’authentification, la sécurité applicative et l’accès à la base de données.

\subsection{Base de données : PostgreSQL via Supabase}
Les données de l’application (utilisateurs, salons, messages, vidéos, playlists, votes, etc.) sont stockées dans une base \textit{PostgreSQL} hébergée via \textit{Supabase}. Ce choix garantit la robustesse, la cohérence et la persistance des données. Laravel interagit avec cette base à l’aide de son ORM \textit{Eloquent}, facilitant la manipulation des entités applicatives.

\subsection{Communication temps réel : Laravel Reverb et Laravel Echo}
Les fonctionnalités temps réel (chat instantané, synchronisation de lecture vidéo, mise à jour des sondages) reposent sur l’utilisation de WebSockets. \textit{Laravel Reverb} agit comme serveur de communication temps réel, tandis que \textit{Laravel Echo} est utilisé côté client pour écouter les événements diffusés. Cette combinaison permet une communication bidirectionnelle fluide et instantanée entre les utilisateurs d’un même salon.

\subsection{API externe : YouTube IFrame API}
L’API \textit{YouTube IFrame} est utilisée pour intégrer le lecteur vidéo directement dans l’interface de l’application. Elle permet de contrôler la lecture (lecture, pause, changement de vidéo) tout en garantissant une compatibilité et une performance optimales.

\subsection{Sécurité : JSON Web Token (JWT)}
L’authentification repose sur l’utilisation de \textit{JSON Web Tokens}. Après connexion, chaque utilisateur reçoit un jeton signé qui est transmis lors des requêtes suivantes afin de garantir un accès sécurisé aux ressources protégées de l’application.

\subsection{Outils collaboratifs}
\begin{itemize}
  \item \textbf{GitHub} : gestion du code source et versioning collaboratif ;
  \item \textbf{Jira / Notion} : planification des tâches, suivi d’avancement et documentation ;
  \item \textbf{Postman} : tests des routes de l’API ;
  \item \textbf{Figma} : conception des maquettes et des diagrammes (UML, interfaces).
\end{itemize}

\section{Architecture logicielle}

L’architecture de \textbf{WithYou} repose sur une séparation claire entre le client, le serveur et la couche de données, favorisant la modularité, la scalabilité et la maintenabilité du système.

\subsection{Front-end (Client React)}
Le front-end est une application React composée de composants réutilisables organisés par fonctionnalités. Il communique avec le back-end via une API RESTful pour les opérations classiques (authentification, création de salons, gestion des playlists, votes, etc.).  
Pour les interactions temps réel, \textit{Laravel Echo} établit une connexion WebSocket avec le serveur afin de recevoir instantanément les événements liés aux salons (messages, synchronisation vidéo, changements d’état).

\subsection{Back-end (Serveur Laravel)}
Le serveur \textit{Laravel} expose une API RESTful sécurisée par JWT. Il gère les utilisateurs, les salons, les playlists, l’historique et les sondages à l’aide de l’ORM \textit{Eloquent}. Les événements temps réel (lecture, pause, changement de vidéo, messages) sont émis via \textit{Laravel Reverb} et diffusés à l’ensemble des clients connectés à un même salon.

\subsection{Base de données (PostgreSQL via Supabase)}
PostgreSQL assure  le stockage persistant des données de manière fiable et sécurisée. \textit{Supabase} fournit l’infrastructure de la base ainsi que des mécanismes de sécurité et de gestion des accès, garantissant la protection des informations sensibles.

\subsection{Communication temps réel (Laravel Reverb et Echo)}
Laravel Reverb agit comme serveur de communication temps réel en gérant les connexions WebSocket et la diffusion des événements. \textit{Laravel Echo} permet au client d’écouter ces événements et de mettre à jour l’interface utilisateur de manière instantanée.

Cette architecture garantit une séparation claire des responsabilités entre les différentes couches du système, facilite l’évolution indépendante de chaque composant et simplifie la maintenance globale de l’application.

\chapter{Gestion de projet}

\section{Méthodologie}

Pour développer la plateforme \textbf{WithYou}, nous avons suivi une démarche structurée de type cascade, en passant successivement par les phases d’analyse des besoins, de conception, de planification, de développement et de tests, tout en travaillant de manière collaborative.

La phase d’analyse a permis d’identifier les fonctionnalités attendues, les contraintes techniques et les objectifs du projet. Elle a servi de base pour définir un périmètre clair et éviter les dérives fonctionnelles.

La phase de conception a consisté à définir l’architecture générale du système (séparation front-end / back-end / base de données), ainsi que la structure des principales entités (utilisateurs, salons, messages, vidéos, playlists, votes).

La planification a ensuite permis de répartir les tâches entre les membres de l’équipe et d’ordonner leur réalisation en tenant compte des dépendances techniques entre les différentes parties du projet.

La phase de développement a été menée de manière progressive, chaque membre implémentant ses fonctionnalités, avec des intégrations régulières afin de maintenir une version fonctionnelle du projet.

Enfin, la phase de tests a permis de vérifier le bon fonctionnement de l’application, de corriger les erreurs détectées et d’améliorer la stabilité globale, en particulier pour les fonctionnalités temps réel et la gestion des sessions.

Cette organisation nous a permis de structurer efficacement notre travail et d’aboutir à une application cohérente et fonctionnelle.

% -------------------------
\section{Cahier des charges et priorisation}

En début de projet, nous avons rédigé un cahier des charges recensant l’ensemble des fonctionnalités envisagées, puis nous les avons classées par ordre de priorité afin de garantir la livraison d’un produit fonctionnel dans les délais impartis.

Nous avons distingué trois niveaux de priorité :

\subsection*{Fonctionnalités essentielles (VIT — Vitales)}
Ces fonctionnalités constituent le cœur du système et sont indispensables à son fonctionnement :
\begin{itemize}
  \item Inscription et authentification sécurisée des utilisateurs
  \item Création et accès aux salons (publics ou privés)
  \item Invitation via lien ou code d’accès
  \item Lecture de vidéos YouTube via l’API officielle
  \item Synchronisation des actions principales (lecture, pause, changement de vidéo)
  \item Chat textuel en temps réel intégré au salon
  \item Gestion de base de la playlist (ajout, suppression, sélection)
\end{itemize}

\subsection*{Fonctionnalités importantes (IMP — Importantes)}
Elles enrichissent l’expérience utilisateur sans être strictement nécessaires au fonctionnement minimal :
\begin{itemize}
  \item Gestion des rôles et permissions dans un salon
  \item Système de sondage pour le choix des vidéos
  \item Notation du salon et de l’expérience utilisateur
  \item Profil utilisateur
  \item Améliorations de l’interface utilisateur
\end{itemize}

\subsection*{Fonctionnalités optionnelles (MIN — Minimales)}
Elles constituent des améliorations complémentaires :
\begin{itemize}
  \item Abonnement premium
  \item Multilingue
  \item Chat vocal
  \item Recommandation automatique de vidéos
  \item Mode sombre
\end{itemize}

Cette priorisation nous a permis de concentrer nos efforts sur les fonctionnalités essentielles et d’assurer la livraison d’un \textit{MVP} (Minimum Viable Product) fonctionnel à la fin du semestre.

\chapter{Problèmes rencontrés et solutions}

Au cours du projet, l’équipe a rencontré plusieurs difficultés, tant sur le plan technique qu’organisationnel. Les principaux problèmes identifiés ainsi que les solutions apportées sont présentés ci-dessous.

\paragraph{Configuration de Laravel Reverb et des WebSockets}
Le serveur WebSocket ne se connectait pas correctement au front-end et les événements diffusés n’étaient pas reçus.  
\textbf{Solution :} configuration des variables d’environnement (ports, clés), intégration de Laravel Echo avec le protocole Pusher et mise en place de tests progressifs.

\paragraph{Synchronisation vidéo}
Les événements de lecture (lecture, pause, changement de vidéo) étaient parfois perdus ou reçus avec un décalage.  
\textbf{Solution :} implémentation d’un événement dédié \texttt{VideoUpdated}, ajout de logs pour tracer les échanges et création de pages HTML de test pour valider les WebSockets.

\paragraph{Problèmes CORS}
Les requêtes entre le front-end (port 5173) et le back-end (port 8000) étaient bloquées par la politique de sécurité du navigateur.  
\textbf{Solution :} configuration du fichier \texttt{cors.php} afin d’autoriser les origines locales et activation du middleware CORS.

\paragraph{Envoi d’e-mails de vérification}
Les e-mails de confirmation n’étaient pas envoyés correctement.  
\textbf{Solution :} configuration des paramètres SMTP dans le fichier \texttt{.env} et utilisation d’un service de test fiable (Mailtrap).

\paragraph{Types de données PostgreSQL}
Des erreurs SQL sont apparues concernant les UUID et les horodatages automatiques.  
\textbf{Solution :} désactivation des timestamps sur certains modèles (\texttt{public \$timestamps = false}) et utilisation correcte de \texttt{Str::uuid()}.

\paragraph{Intégration front-end / back-end}
Les formats de données entre les mocks front-end et l’API réelle n’étaient pas cohérents.  
\textbf{Solution :} définition claire des contrats d’API, typage TypeScript des interfaces et synchronisation régulière entre les équipes.

\paragraph{Conflits Git et gestion des versions}
Des conflits sont apparus lors des fusions et certains commits ont été perdus.  
\textbf{Solution :} adoption de bonnes pratiques Git (branches de fonctionnalités, commits fréquents, \texttt{git pull --rebase}) et sessions de sensibilisation au sein de l’équipe.

\paragraph{Retard initial}
Une phase d’apprentissage des technologies (Laravel, React, WebSockets) a ralenti le démarrage.  
\textbf{Solution :} réorganisation du planning en sprints courts et priorisation du MVP (Minimum Viable Product).

\subsection*{Mesures préventives mises en place}

\begin{itemize}
  \item Réunions hebdomadaires d’avancement
  \item Documentation des solutions (README, Notion)
  \item Backlog partagé (Jira / Notion)
  \item Pages de test HTML pour valider les fonctionnalités isolément
  \item Revue de code avant chaque fusion
\end{itemize}

\chapter{Tests et validation}

Conformément à la méthodologie en cascade adoptée, les phases de test ont été réalisées principalement en fin de développement, après l’implémentation des fonctionnalités principales. Les tests ont été effectués essentiellement de manière manuelle.

\paragraph{Tests fonctionnels manuels}
Chaque fonctionnalité clé (authentification, création de salons, playlist, chat, sondages, notation, historique) a été testée pas à pas afin de vérifier la cohérence des parcours utilisateurs, la gestion des erreurs (champs manquants, identifiants invalides, codes erronés) et la persistance des données dans PostgreSQL / Supabase.

\paragraph{Tests d’intégration API}
Les routes back-end ont été testées à l’aide de Postman et de pages HTML dédiées afin de vérifier les réponses JSON, l’authentification JWT et la cohérence des échanges entre le front-end et le back-end.

\paragraph{Tests de communication temps réel}
La synchronisation vidéo et le chat ont été testés en ouvrant plusieurs sessions simultanées afin de valider la réception des événements WebSocket, y compris dans des situations limites (déconnexion, reconnexion, utilisateurs multiples).

\paragraph{Limites et perspectives}
L’absence de tests automatisés a rendu le débogage plus long en fin de projet. La synchronisation vidéo reste partiellement stabilisée et nécessite encore des ajustements. Une approche plus itérative et l’ajout de tests unitaires et d’intégration automatisés permettraient d’améliorer la fiabilité et la maintenabilité de l’application.

Le projet \textbf{WithYou} constitue une plateforme collaborative fonctionnelle et cohérente.
